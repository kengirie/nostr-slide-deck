# Nostr スライド共有サービス 設計資料（ドラフト）

Speaker Deck のようなスライド共有サービスを Nostr の上に作るための設計メモ。
検討の経緯と、たどり着いた結論をまとめる。

---

## 1. 何を作るか

発表スライド（PDF）をアップロードすると、きれいな閲覧ページと共有用リンクができて、
X や Slack に貼るとスライド1枚目のカード画像つきで展開される。
見た人はコメントや「いいね」ができて、発表者に zap（ビットコインの少額投げ銭）を直接送れる。

ひとことで言うと **「投げ銭できる Speaker Deck」**。

### なぜ Nostr で作るのか

- **アカウントを作らせなくていい。** Nostr の鍵を持っている人は、そのままの名前・フォロワーで使い始められる。
- **ソーシャル機能を作らなくていい。** コメント・リアクション・フォロー・投げ銭は、Nostr の共通仕様（NIP）に乗るだけで手に入る。しかも他のアプリからも見える。
- **運営が消えてもデータが残せる。** スライドとメタ情報は運営のデータベースではなく、公開されたリレーとファイルサーバーに置かれるため。

### 現状の空白

調べた限り、Nostr にスライド共有の定番アプリはまだ無い。
「Slidestr」という名前のアプリはあるが、これは画像・動画をスライドショー表示で眺めるメディアビューアで、プレゼン資料の共有とは別物。ニッチは空いている。

---

## 2. 設計の合言葉：賢いクライアント、馬鹿なサーバー

Nostr の思想は「サーバーゼロ」ではなく **「サーバーは交換可能な土管であれ」**。
サーバーはこのアプリのことを何も知らず、落ちたら別のサーバーに差し替えられる。
アプリ固有の賢い処理は、すべて利用者の手元（ブラウザ）で行う。

この方針から導かれる最重要ルールがこれ：

> **「リクエストが来たときに何かを作る」処理を持たない。**
> **作れるものは、公開の瞬間にクライアントが全部作り終えておく。**

スライドは公開後ほとんど変わらないコンテンツなので、この方針と非常に相性がいい。

## 3. 登場人物（使う部品）

| 部品 | 平たく言うと | 役割 |
|---|---|---|
| **リレー** | 掲示板サーバー | 小さな JSON のメモ（イベント）を預かって配る。スライドの「目録」を流す場所 |
| **Blossom** | 荷物倉庫 | PDF・画像・HTML などのファイル本体を預かる。ファイルの住所は中身のハッシュ値（≒指紋）。同じ指紋なら、どの倉庫から取り出しても同じファイル。預け入れは Nostr の鍵で署名して認証 |
| **nsite** | 静的サイトの公開手段 | 「このパスにはこの指紋のファイル」という対応表をイベントとして公開し、サイト一式を Blossom に置く仕組み |
| **ゲートウェイ** | 配達口 | nsite を `https://<npub>.nosto.re/...` のような普通の URL で配るだけのサーバー。中身を知らない土管。別のゲートウェイに差し替え可能 |
| **インデクサ** | 図書館の司書 | 検索・新着・人気ランキング用の集計サーバー。唯一少し賢いが、読み取り専用のキャッシュにすぎず、消えても閲覧・公開は成立する |

サーバー側に、このアプリ専用のプログラムは**一つも無い**。すべて汎用の部品。

## 4. データ設計

### 4.1 デッキイベント（スライド1本の目録）

スライド1本につき、1つの「差し替え可能イベント」（addressable event、kind 30000〜39999 の未使用番号をひとつ決める）を発行する。
同じ識別子で発行し直せば上書きになるので、スライドの差し替え（v2 公開）が自然にできる。

イベントに入れる情報（タグ）：

- `d` … デッキの識別子（上書きの鍵になる）
- `title` / `summary` … タイトルと概要
- `image` … サムネイル（1ページ目）の URL
- ファイル情報 … PDF 本体の URL・ハッシュ・MIME タイプ・サイズ。書き方は NIP-94（ファイルメタデータの共通仕様）の流儀に合わせる
- ページ画像のリスト … 各ページ画像のハッシュを順番どおりに
- `t` … ハッシュタグ（発見用）

ポイントは、**URL とハッシュを両方書く**こと。URL は「今どこにあるか」、ハッシュは「何であるか」。
URL の先の倉庫が消えても、ハッシュを頼りに別の倉庫から同じファイルを引ける。

### 4.2 ソーシャル機能は既製品

自作しない。すべて Nostr の共通仕様に乗る。

- コメント … NIP-22（あらゆるイベントにぶら下げられるコメント）
- いいね … NIP-25（リアクション）
- 投げ銭 … NIP-57（zap）
- プロフィール・フォロー … 既存のものがそのまま使える
- アプリの名乗り … NIP-89 に登録しておくと、他の Nostr アプリでデッキイベントを見た人が「このアプリで開く」と誘導される

## 5. 公開の流れ（すべてブラウザの中で完結）

1. 利用者が PDF をドロップする
2. ブラウザ内で PDF を1ページずつ画像に変換する（pdf.js）。1ページ目からサムネイルも作る
3. PDF 本体と画像一式を Blossom 倉庫に預ける（**2〜3か所に同じものを預けて**冗長化する）
4. **デッキ専用の閲覧ページ（HTML）をその場で生成する。** この HTML には `og:title` や `og:image` などの meta タグと、ページ送りビューアの JS を焼き込んでおく
5. HTML 一式を nsite として公開する（パス→指紋の対応表イベントを発行）
6. デッキイベント（4.1）をリレーに発行する

サーバー側で動く処理：**ゼロ**。

---

## 6. 閲覧とシェア（OG 画像問題の解決）

### 6.1 問題

Speaker Deck の強さの半分は「リンクを貼ると1枚目がカードで展開される」こと。
ところが X や Slack のクローラーは JavaScript を実行しないので、
ふつうの SPA（JS で画面を組み立てるサイト）では OG カードが出ない。
素直に作るとサーバーサイドレンダリング（SSR）が必要になり、「馬鹿なサーバー」の方針に反する。

### 6.2 答え：レンダリングを「リクエスト時」から「公開時」に移す

疑うべき前提は「meta タグはリクエストが来たときに生成するもの」だった。
スライドはほぼ不変なのだから、**公開の瞬間にクライアントが meta タグ入りの完成品 HTML を作って倉庫に置いておけばいい**（＝手順5の4）。

- 人間がリンクを開く → ゲートウェイが完成品 HTML を返す → ビューアが動く
- クローラーが同じリンクを叩く → 同じ完成品 HTML が返る → OG カードが出る（JS 不要）
- Nostr アプリ内で見る → デッキイベントを直接読んで表示（HTML を経由しなくてよい）

SSR は不要になった。

### 6.3 og:image の注意点

`og:image` に特定の Blossom 倉庫の URL を直書きすると、その倉庫が消えた瞬間に共有カードが壊れる。
`og:image` もゲートウェイ経由の URL（ゲートウェイが指紋から生きている倉庫を探して返す）にしておく。
ここで初めて「指紋が住所」という Blossom の性質が OG カードの生存性にも効いてくる。

---

## 7. あえてやらないこと

- **SSR・動的な meta タグ生成** … 6.2 により不要
- **独自アカウント・独自ログイン** … Nostr の鍵がアカウント
- **独自のコメント・いいね・課金システム** … NIP に乗る
- **独自ストレージ API** … Blossom に乗る

「作らないもの」がこれだけ多いのが、Nostr で作る最大の利点。

---

## 8. 割り切りと残る課題

- **永続性はタダではない。** Blossom 倉庫も消えるし、保管には費用がかかる。ハッシュのおかげで「引っ越しやすい」だけで、「勝手にずっと残る」わけではない。対策は複数倉庫へのミラーと、（将来的には）有料の長期保管オプション。
- **URL の見た目がゲートウェイのドメインに依存する。** `nosto.re` などの汎用ゲートウェイで始め、必要なら独自ドメインのゲートウェイを立てる（それも土管なので誰でも代替を立てられる）。
- **検索・トレンドはインデクサ頼み。** リレーは検索が苦手。インデクサが無くても閲覧はできるが、発見性は落ちる。MVP 後の課題。
- **限定公開・社外秘スライドはスコープ外。** すべて公開が前提。将来やるなら NIP-44（暗号化）方向の検討になるが、公開型の価値とは別物なので急がない。

---

## 9. 最小版（MVP）の範囲

これだけ動けば成立する：

1. PDF をドロップすると公開される
2. リンクを X / Slack に貼ると1枚目のカードが出る
3. ページ送りで閲覧できる
4. zap で投げ銭できる

コメント・検索・ランキング・差し替え UI は、すべてその後でいい。

---

## 10. まとめ

| 従来の Web での作り方 | この設計 |
|---|---|
| アプリサーバー + DB + ストレージ + SSR | 汎用の土管（リレー / Blossom / ゲートウェイ）だけ |
| リクエストが来たら HTML を作る | 公開の瞬間に HTML まで作り終える |
| アカウント・コメント・課金を自作 | Nostr の共通仕様に乗る |
| 運営が畳んだらデータ消滅 | 指紋を頼りに誰でもミラー・移住できる |

スライドという「公開後ほぼ不変のコンテンツ」は、この作り方に最も向いた題材のひとつ。
